Process Flow · 10 Steps
Production Floor Execution
Process Order release through TECO — with RCA branches at every failure point
Process step
Checkpoint — verify before proceeding
RCA branch — failure investigation
Order close / settlement
1
Process Order Released
COR2
Planner changes status to REL. This is the ignition — it activates reservations and signals the warehouse that components are needed. Orders are typically created in advance via MRP and released on or near the production day.
2
Reservations Auto-Generated
MB26
SAP auto-creates reservations for all BOM components when the order is created or released. These are soft holds — they signal demand but don't move stock. View open reservations in MB26 filtered by order or material.
3
Warehouse Tasks Assigned
CO27 / LT0A
Transfer Orders and worklist items are generated. Warehouse team sees what needs to be staged. CO27 is the component staging worklist for Process Orders — the primary tool in food manufacturing. LT0A for WM-managed TOs.
4
⚑ Checkpoint
Components Staged to Line
MB26
Physical staging. Verify material number, quantity, and unit of measure against the reservation before issuing. Any mismatch here compounds downstream. Do not allow staging of QI or blocked stock.
⚡ RCA — Staging Error
Root Cause Questions
  • Is the BOM quantity correct? (CS03)
  • Was the right reservation used?
  • UoM mismatch — cases vs. lbs?
  • Wrong material pulled from shelf?
Resolution Path
  • Correct staging physically before GI
  • If GI already posted: reverse MT 262
  • BOM error → escalate to MM/PP team
  • Document in IC log with reason code
5
Goods Issue Posted — Components
MIGO · MT 261
Components are formally issued to the Process Order. RM stock decreases. Cost charged to order. In many plants this happens via backflush automatically during production confirmation — if backflush is active, a manual MT 261 should not also be posted (double-posting risk).
6
Production Execution
Floor
Physical manufacture. Components are consumed, salsa is produced. SAP is not directly involved at this step — but any deviation (recipe change, unexpected scrap, rework) must be documented here to be recorded correctly in the next steps.
7
⚑ Checkpoint
Yield Check
COR3
Compare actual output quantity to Process Order expected quantity before posting GR. Any significant variance should be investigated now — once GR is posted and order is settled, tracing the root cause becomes significantly harder.
⚡ RCA — Yield Variance
Root Cause Questions
  • Was scrap recorded during run?
  • Did recipe deviation occur?
  • Scale/weigh calibration accurate?
  • Was a batch reworked without posting?
  • Was overissue of components posted?
Resolution Path
  • Post scrap entry if applicable (MT 551)
  • Record rework batch in order confirmation
  • Recipe deviation → escalate to QA
  • Overissue → reverse with MT 262
8
GR of Finished Goods
MIGO · MT 101
Finished goods are received into FG storage location against the Process Order. FG stock increases. SAP values the GR at standard cost and begins tracking the variance between actual costs and standard. Must match confirmed yield quantity.
9
⚑ Checkpoint
Putaway & Stock Verification
MMBE
Confirm FG stock increased in the correct storage location. Verify in MMBE immediately after posting. Common error: GR posted to RM storage location instead of FG location — caught here before cycle count finds it.
⚡ RCA — Wrong Putaway Location
Root Cause Questions
  • Did GR post to correct storage location?
  • Was wrong location selected in MIGO?
  • Does material master default to wrong location?
Resolution Path
  • Reverse GR with MT 102
  • Repost MT 101 to correct location
  • Or transfer via MT 311 (loc → loc)
10
Order Confirmed & TECO
COR2 → KO88
TECO Technically Complete — closes order to further movements. Does not require variances to be resolved beforehand. Settlement run (KO88 individual / CO88 mass) then posts variance to Finance. Investigate large variances before TECO — tracing root cause post-settlement is significantly harder.
Process Flow · 8 Steps
Inbound Receiving
Purchase Order through Invoice Verification — raw material receipt with RCA branches
Process step
Checkpoint — verify before proceeding
RCA branch
1
Purchase Order Exists
ME23N
Created by Purchasing in ME21N. As IC Lead, use ME23N to view and validate the PO before any truck arrives — confirm material number, expected quantity, unit of measure, and vendor. Never receive against a PO you haven't reviewed.
2
⚑ Checkpoint — At Dock
Delivery Arrives — Physical Verification
ME23N
Before any SAP posting: verify material, quantity, condition, and vendor against the PO. This is the golden window — catching discrepancies here prevents all downstream document chain issues. Once MT 101 is posted, corrections require reversals.
⚡ RCA — Dock Discrepancy
Common Discrepancies
  • Wrong material delivered
  • Short shipment (fewer units than PO)
  • Damaged or out-of-spec goods
  • PO quantity mismatch
Resolution Path
  • Short ship: post partial GR for actual qty
  • Wrong material: do NOT post. Contact Purchasing
  • Damaged: post MT 103 (blocked stock)
  • QA accepts: MT 105. Reject: MT 122
3
Post Goods Receipt
MIGO · MT 101
Reference the PO number. System creates a Material Document (inventory record) and an Accounting Document (debit inventory, credit GR/IR). Post for the quantity actually received, not the PO quantity if they differ. Same-day posting is non-negotiable.
4
⚑ Checkpoint — Post-Posting Validation
Validate Material Document
MB03 / MB51
Immediately after posting, verify the material document: quantity, unit of measure, storage location, and batch (if applicable). Use MB03 for individual doc or MB51 for daily list filtered to MT 101.
⚡ RCA — Posting Error
Root Cause Questions
  • Was UoM correct? (cases vs. lbs)
  • Correct storage location selected?
  • Was correct PO line referenced?
  • Was quantity entered correctly?
Resolution Path
  • Reverse immediately with MT 102
  • Repost with corrected values
  • Document reason in IC log
5
⚑ Checkpoint — Stock Confirmation
Confirm Stock Updated in IM
MMBE
Run MMBE for the received material and confirm unrestricted stock increased in the correct storage location. Cross-reference with physical count on the floor. This is your reconciliation before the material leaves your sight.
6
Physical Putaway
Physical
Material is stored in the designated location. Material is now available for reservations. If location differs from what was posted in MIGO, resolve via MT 311 transfer before production needs it.
7
Invoice Verification
MIRO (AP)
AP posts 3-way match: PO price / GR quantity / vendor invoice. Your GR accuracy directly determines whether this step flows or blocks. If invoice is blocked, the root cause almost always traces back to Step 3 or 4. AP will come to IC — have your MB51 ready.
⚡ RCA — 3-Way Match Failure
Root Cause Questions
  • GR qty ≠ invoice qty?
  • GR not yet posted?
  • PO price different from invoice price?
  • Wrong PO referenced on invoice?
Resolution Path
  • Missing GR → post MT 101 immediately
  • Wrong qty → reverse MT 102, repost
  • Price variance → Purchasing updates PO
  • Wrong PO → AP corrects in MIRO
8
⚑ IC Daily Routine
Open GR Daily Check
MB51
Run every day end-of-shift: MB51 → filter Movement Type 101 → today's date. Any physical receipt that occurred today must appear here. Unposted GRs = phantom shortage in production + blocked AP invoice. This single check prevents the majority of receiving discrepancies.
Process Flow · 8 Steps
Outbound Shipping
Sales Order through Billing — finished goods shipment with RCA branches
Process step
Checkpoint — verify before proceeding
RCA branch
1
Sales Order Exists
VA03
Created by Sales in VA01. Drives MRP demand. View in VA03 to confirm material, quantity, customer, and requested delivery date before any warehouse action begins.
2
Outbound Delivery Created
VL01N
Delivery document generated referencing the SO. This is the shipping instruction document. Triggers the picking requirement. In an IM-only environment, no system TOs are generated — picking is a physical act against the delivery document.
3
⚑ Checkpoint — Before Picking
Stock Availability Check
MMBE
Verify FG stock in MMBE before any picking begins. A shortage discovered after the truck is loaded is a crisis. Discovered at Step 3, it's a planning conversation.
⚡ RCA — Stock Shortage
Root Cause Questions
  • Was FG GR posted from production?
  • Is FG in wrong storage location?
  • Was a prior GI posted in error?
  • Is stock in QI or blocked status?
Resolution Path
  • MMBE — check all storage locations
  • MB51 MT 101 — confirm FG GR was posted
  • Transfer from correct location (MT 311)
  • Escalate to production if truly short
4
Picking
VL06O
Physical pull from FG storage location. In IM-only, no system TO is generated — warehouse team physically picks against the delivery document. Monitor open deliveries in VL06O.
5
⚑ Checkpoint — Before Loading
Load Verification
Delivery Doc
Before the truck door closes: verify material number, quantity, and customer on the physical load matches the delivery document exactly. This is the last correction window before GI is posted. Wrong SKU or short load discovered here = easy fix. Discovered after GI = reversal required.
⚡ RCA — Load Error
Root Cause Questions
  • Wrong SKU picked from floor?
  • Short load — fewer cases than delivery?
  • Wrong customer pallet mixed in?
Resolution Path
  • Stop and correct before posting GI
  • Amend delivery document if needed
  • Only post GI after load is verified
6
Post Goods Issue
VL02N · MT 601
This is the transaction that removes FG from book inventory. Must be posted the same moment the truck leaves — or before. Creates the accounting document (debit COGS, credit inventory). Unlocks billing. A delivery that exists but has no GI post is the #1 source of phantom FG stock.
7
⚑ IC Daily Routine
Open Delivery Daily Check
VL06O
Run end-of-shift: VL06O → filter open deliveries where GI not posted. Any delivery older than same-day without a GI post is a risk. If truck left, post GI immediately. If in question, physically verify before posting.
⚡ RCA — GI Never Posted
What Happens Without GI Post
  • Book shows FG that physically left
  • Billing is blocked — customer not invoiced
  • Cycle count will find phantom surplus
  • AP/Finance will escalate
Resolution Path
  • VL06O → find open delivery
  • Confirm truck left (driver manifest)
  • Post GI via VL02N immediately
  • If truck returned: MT 602 reversal first
8
Billing Document Created
VF01 (Finance)
Finance creates customer invoice in VF01. Fully dependent on GI being posted. Billing due list visible in VF04. If blocked, root cause is almost always a missing or incorrect GI — trace back to Step 6.
Process Flow · 7 Steps
Cycle Count Program
Fiori-based physical inventory — scheduled cycle count through variance analysis
Process step
Checkpoint / decision
Monthly analysis
⚑ Prerequisite: Cycle count indicators (A/B/C/D) must be set on material masters in MM02 → Plant Data / Storage 1 → CC Phys. Inv. Ind. Without these, the "Create Physical Inventory… Cycle Counting" Fiori tile returns no materials. Request mass update via your SAP team using MM17.
1
Generate Count Documents
Fiori: Create PI… Cycle Counting
Enter Plant 1730 and storage location. SAP compares every material's last count date against its A/B/C/D indicator frequency and auto-creates PI documents for all materials due. This replaces manual MI01 creation entirely. Run weekly.
2
Distribute to Counters
Fiori: Print PI Documents
Print count sheets without system quantities visible (blind count). Preferred: assign counters direct Fiori tablet access for real-time floor entry — eliminates paper and transcription errors. Assign by storage location zone.
3
⚑ Checkpoint — Blind Count
Physical Count (Blind)
Fiori: Count PI (List)
Counters must not see book quantity before entering their count — this is a basic audit control. If system is showing book quantities, flag to your SAP team as a configuration issue. Blank entries are acceptable if unsure — never guess. Use the List view for batch efficiency across multiple materials.
4
⚑ Checkpoint — Supervisory Review
Review Variances
Fiori: Process PI Count Results
Do not post immediately. Review all variances against a tolerance threshold. For each variance above tolerance, ask: Was there a GI or GR posted today that may have moved the book? Is the unit of measure correct? Did the counter count the right material? Investigate before posting — posted variances create accounting entries.
⚡ RCA — Count Variance Investigation
Root Cause Questions
  • Unposted GI or GR from today?
  • Wrong UoM counted? (cases vs. lbs)
  • Counter counted wrong material?
  • Material in wrong location?
  • BOM over/underissue pattern?
Resolution Path
  • Check MB51 for same-day movements first
  • Trigger recount (Fiori: Change PI Doc)
  • Different counter for recount — never same person
  • Post variance with reason code once confirmed
5
Recount if Needed
Fiori: Change PI Document
Flag suspicious documents for recount. A second, independent counter resets the entry. Never let the original counter recount their own variance. Two independent counts that agree is the standard for posting an unexplained variance.
6
Post Differences
Fiori: Process Physical Inventory
Post the confirmed differences. Always apply reason codes. This is the difference between a count history and a diagnostic dataset. Without reason codes, you have numbers. With reason codes, you have evidence of where your upstream process is breaking down.
7
Monthly Trend Analysis
Fiori: Manage PI Documents
Monthly Review count history filtered by reason code and material. A material that appears in every cycle count is a process problem upstream — not a counting problem. Reason code clusters reveal the failure type:
  • Unposted GI/GR → posting discipline issue in production or receiving
  • Wrong location → putaway process or label clarity issue
  • BOM variance → recipe or master data issue
  • Recurring surplus → possible unreported returns or by-products
IC Reference · Navigation Guide
Data Checkpoints
Every transaction you need to find, compare, and investigate discrepancies — with exact navigation paths
Navigation syntax: /n[TCODE] in the SAP GUI command field jumps directly to any transaction. In Fiori, search the tile name in the launchpad search bar.
🔴 Daily — Run Every Shift
📋
MB51
Material Document List
SAP GUI: /nMB51 → Enter Plant, Movement Type, Date Range → Execute
Your primary audit trail. Shows every goods movement for a material, plant, or movement type over a date range. The single most-used IC transaction.
Look for
  • MT 101 today — any physical receipt without a corresponding document?
  • MT 261 — component issues that don't match BOM quantities
  • MT 601 — GIs posted for deliveries that should still be open
  • MT 501/521 — ad-hoc receipts without PO/Order reference (process gap)
  • Duplicate material documents for same material/date
● Daily
📦
MMBE
Stock Overview
SAP GUI: /nMMBE → Enter Material + Plant → Execute → Expand all levels
Snapshot of current stock across all storage locations, stock types (unrestricted, QI, blocked), and special stocks. Your ground truth for what SAP thinks you have right now.
Look for
  • Stock in wrong storage location (FG in RM location or vice versa)
  • Unexpected QI or blocked stock — items awaiting disposition
  • Zero stock on a material that production just received
  • Negative stock — a posting error that must be resolved immediately
  • Compare unrestricted vs. physical count for spot checks
● Daily
🚚
VL06O
Outbound Delivery Monitor
SAP GUI: /nVL06O → Select "For Goods Issue" → Enter Plant + Date → Execute
Shows all outbound deliveries by status. The critical filter is deliveries where GI has not been posted — these represent trucks that may have physically left but book inventory is still showing the product.
Look for
  • Any delivery created today with no GI post by end of shift
  • Deliveries from yesterday or prior with open GI status
  • Deliveries with picking confirmed but GI missing
  • Deliveries with status errors or processing locks
● Daily
🔍
MB26
Open Reservations
SAP GUI: /nMB26 → Enter Plant + Storage Location + Date → Execute
Shows all open reservations (unfulfilled component demand from Process Orders). Critical for understanding what production needs today and whether stock is available to fulfill it.
Look for
  • Reservations for today's orders with insufficient unrestricted stock
  • Old open reservations for completed or cancelled orders (should be zero'd)
  • Reservations referencing materials in QI or blocked stock
  • Duplicate reservations for same order/component
● Daily

ME2M
Purchase Orders by Material
SAP GUI: /nME2M → Enter Material + Plant → Scope: "Open POs" → Execute
Lists all open POs for a material. Lets you compare expected inbound deliveries against current stock levels — forward-looking IC view.
Look for
  • POs with delivery date passed but no GR posted
  • Open PO quantities that would cause overstock if received
  • POs with partial GR where balance is still open unexpectedly

SAP GUI: /nCO27 → Enter Plant + Production Date Range → Execute
Component staging worklist for Process Orders. Shows what has been staged vs. what is still needed. Used by warehouse team but IC Lead should review weekly to spot systematic staging gaps.
Look for

  • ⚑Orders staging from QI stock (should not happen)
  • ⚑Repeated staging failures on same material

MB52
Warehouse Stocks of Material
SAP GUI: /nMB52 → Enter Plant + Storage Location → Execute
Lists all materials with stock in a given storage location, including value. Broader than MMBE — use for a full inventory snapshot of a location rather than a single material lookup.
Look for
  • Materials with stock in locations they shouldn't be in
  • Zero-stock materials that had activity this week
  • High-value materials for prioritizing A-indicator cycle counts
● Weekly
🔄
COR3
Display Process Order
SAP GUI: /nCOR3 → Enter Order Number → Review Components tab + Quantities tab
Full detail view of a Process Order — planned vs. actual quantities for components and finished goods. Essential for investigating yield variances and consumption discrepancies before TECO.
Look for
  • Actual GI qty vs. planned component qty — overissue or underissue
  • Actual GR qty vs. order confirmed qty — yield gap
  • Orders in REL status past their scheduled end date
  • Orders with zero GR qty but GI already posted — production never completed
● Weekly
🟢 Monthly — Trend & Analysis
📈
MI24
Physical Inventory Differences
SAP GUI: /nMI24 → Enter Plant + Fiscal Year → Execute
Historical report of all posted cycle count variances. The most powerful IC analytical tool — reveals which materials consistently have discrepancies, which reason codes are most common, and the dollar value of inventory adjustments.
Look for
  • Same material appearing in multiple count periods — systemic process issue
  • Dominant reason codes — identifies upstream failure type
  • High-value variance materials — need more frequent cycle count
  • Trend direction — variances growing or shrinking over time?
● Monthly
💰
MB5L
Stock Values by Account
SAP GUI: /nMB5L → Enter Plant + Posting Date → Execute
Reconciles book inventory value at the IM level against the G/L accounts. Finance uses this for period close — IC uses it to confirm that inventory postings align with what Finance is seeing. Discrepancies here signal a posting configuration or valuation issue.
Look for
  • IM value ≠ G/L value — indicates unposted or mis-valued movements
  • Large unexplained variance accounts
  • Negative stock values (always an error)
● Monthly
🗂️
MB5B
Stocks for Posting Date
SAP GUI: /nMB5B → Enter Plant + Key Date → Execute
Shows what stock looked like on any specific past date. Use for end-of-month reconciliation, investigating when a discrepancy first appeared, or auditing stock levels at period close.
Look for
  • Compare end-of-month stock to finance reports
  • Identify when a discrepancy first appeared by comparing periods
  • Reconcile physical annual inventory against system at key date
● Monthly
⚠️
MB5M
Expiration Date / Shelf Life
SAP GUI: /nMB5M → Enter Plant + Expiry Date Range → Execute
Lists all batches approaching or past their expiry date. Critical for food manufacturing — expired product still showing as unrestricted stock is both a food safety risk and a phantom stock problem.
Look for
  • Batches past best-by with unrestricted status — must scrap (MT 551) or transfer to blocked
  • RM batches expiring within 30 days — alert production planning
  • FG batches approaching customer shelf-life requirements
● Monthly

MIGO Movement Type Reference
Every movement type with reversal counterpart and real-world troubleshooting scenario
Rule: The reversal is always the next even number. MT 101 → reversed by MT 102. MT 261 → reversed by MT 262. Always reference the original material document number when reversing.

GR/PO

Reverse
GR against Purchase Order or Production Order
Scenario: Receiving team posts GR for 500 cases of tomato paste but only 450 physically arrived. Book now shows 500. → MB51 filter MT 101 same day. Reverse with MT 102. Repost for actual 450. Contact Purchasing re: short shipment.
103
GR Blocked
104
Reverse
GR into GR Blocked Stock — not valuated
Scenario: Jalapeño delivery arrives with torn packaging — quality uncertain. Cannot accept into unrestricted. → Post MT 103 (blocked). QA inspects. Accept: MT 105 releases to unrestricted. Reject: MT 124 returns to vendor.

Release
106
Reverse
Release GR Blocked Stock to Unrestricted
Scenario: QA clears jalapeños from MT 103 receipt. Need to release for production reservations. → Post MT 105 referencing original MT 103 document. Stock moves to unrestricted and becomes visible in MB26.
122
Return
123
Reverse
Return Delivery to Vendor from Unrestricted Stock
Scenario: 200 lbs of onion powder accepted, but lab finds moisture content out-of-spec 3 days later. → Post MT 122 ref. original PO GR. Stock reduces. GR/IR clears. AP issues debit memo to vendor.
161
Returns PO
162
Reverse
GR for Returns Purchase Order
Scenario: Vendor sends replacement for previously returned batch. A dedicated Returns PO was issued by Purchasing. → Post MT 161 against the Returns PO (not original). Clean audit trail separate from original PO.

Reverse

521
GR no Order
522
Reverse
GR without Production Order Reference
Scenario: FG produced during system outage. Process Order could not be confirmed. → Post MT 521 as emergency. Then post MT 261 manually against order once system restored. ⚠ Double-posting risk — always reconcile after.

Init. Stock
562
Reverse
Initial Entry of Stock Balances
Scenario: New material master created for new sauce SKU. Physical count shows 300 cases on hand but no SAP history. → Post MT 561 for opening balance. ⚠ ONLY for go-live or new material setup. NEVER use 561 to correct a discrepancy — use a cycle count PI document instead.

GI Order
262
